Razor Pages 基本概念

理解以頁面為中心的 ASP.NET Core 前端架構
學習目標
  • 理解 Razor Pages vs MVC Controller 的差異
  • 理解 PageModel 的角色與職責
  • 理解 OnGet / OnPost handler 命名慣例
核心概念
💡 Razor Pages 每個頁面由兩個檔案組成：.cshtml（畫面）和 .cshtml.cs（PageModel，邏輯）。 不需要 Controller，路由由檔案路徑決定，適合以頁面為中心的應用。 例如 Pages/Cart.cshtml 自動對應 /Cart 路由。
Razor Pages vs MVC 對照表
特性ASP.NET Core MVCRazor Pages
處理邏輯位置 Controller 類別（獨立檔案） PageModel 類別（與頁面同目錄）
Action 方法 任意命名，搭配 [HttpGet] / [HttpPost] 固定命名慣例：OnGet / OnPost
路由設定 需要 [Route] / [HttpGet("path")] 由 @page 指示詞和目錄結構決定
View 綁定 回傳 View(model)，model 是獨立物件 PageModel 本身就是 model，@Model.屬性
適合場景 複雜 API、多 action 的資源操作 以頁面為中心的 UI、電商前端

using Refit;

public interface ICatalogService
{
    // [Get] attribute 對應 HTTP GET，路徑即為 API 端點
    [Get("/catalog-service/products")]
    Task<IEnumerable<ProductModel>> GetProducts();

    [Get("/catalog-service/products/{id}")]

    [Get("/catalog-service/products/category/{category}")]
    Task<IEnumerable<ProductModel>> GetProductsByCategory(string category);
}

public interface IBasketService

}

Program.cs 註冊方式
Program.csC#
var builder = WebApplication.CreateBuilder(args);

// 三個 service 共用同一個 GatewayAddress
// YARP 根據路徑前綴 /catalog-service/ /basket-service/ /ordering-service/ 路由
builder.Services
    .AddRefitClient<ICatalogService>()
    .ConfigureHttpClient(c =>
        c.BaseAddress = new Uri(config["ApiSettings:GatewayAddress"]!));

builder.Services
    .AddRefitClient<IBasketService>()
    .ConfigureHttpClient(c =>
        c.BaseAddress = new Uri(config["ApiSettings:GatewayAddress"]!));

builder.Services
    .AddRefitClient<IOrderingService>()
    .ConfigureHttpClient(c =>
        c.BaseAddress = new Uri(config["ApiSettings:GatewayAddress"]!));

💡 三個 Refit service 都指向同一個 GatewayAddress（YARP API Gateway）。 區分不同後端微服務的方式是 URL 路徑前綴：/catalog-service/* 對應 Catalog.API、 /basket-service/* 對應 Basket.API、/ordering-service/* 對應 Ordering.API。 Shopping.Web 永遠不直接呼叫微服務，一律透過 Gateway。

PageModel 和 Refit 整合

在 PageModel 注入 Refit service，處理 GET/POST 資料流
學習目標
  • 理解如何在 PageModel 注入 Refit service
  • 理解 OnGet / OnPost 的 async 版本
  • 理解 BindProperty 綁定表單資料
IndexModel 完整範例（首頁商品列表）
Pages/Index.cshtml.csC#
public class IndexModel : PageModel
{
    private readonly ICatalogService _catalogService;
    private readonly IBasketService _basketService;

    // DI：ASP.NET Core 自動注入 Refit 產生的實作
    public IndexModel(ICatalogService catalogService, IBasketService basketService)
    {
        _catalogService = catalogService;
        _basketService = basketService;
    }

    public IEnumerable<ProductModel> ProductList { get; set; } = Enumerable.Empty<ProductModel>();

    // OnGetAsync：頁面載入時呼叫 Catalog API 取得商品
    public async Task OnGetAsync()
    {
        ProductList = await _catalogService.GetProducts();
    }

    // OnPostAddToCartAsync：名稱中的 "AddToCart" 對應 asp-page-handler="AddToCart"
    public async Task<IActionResult> OnPostAddToCartAsync(string productId)
    {
        var product = await _catalogService.GetProduct(productId);

        // 取得或建立使用者的購物車
        var basket = await LoadUserBasket();

        // 將商品加入購物車
        basket.Items.Add(new ShoppingCartItemModel
        {
            ProductId = product.Id,
            ProductName = product.Name,
            Price = product.Price,
            Quantity = 1,
            Color = "Black"
        });

        // 儲存購物車並導向 Cart 頁面
        await _basketService.StoreBasket(basket);
        return RedirectToPage("Cart");
    }

    private async Task<ShoppingCartModel> LoadUserBasket()
    {
        try
        {
            // 嘗試取得現有購物車
            return await _basketService.GetBasket("swn");
        }
        catch (ApiException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
        {
            // 找不到購物車時建立新的空購物車
            return new ShoppingCartModel { UserName = "swn" };
        }
    }
}

💡 加入購物車的流程：LoadUserBasket()（取得或建立空購物車） → 加入商品 item → StoreBasket()（寫回 API）→ 導向 Cart 頁面。 這是一個典型的「讀取→修改→寫回」模式（Read-Modify-Write）。
CheckoutModel 範例（BindProperty 表單綁定）
Pages/Checkout.cshtml.csC#
public class CheckoutModel : PageModel
{
    private readonly IBasketService _basketService;

    public CheckoutModel(IBasketService basketService)
    {
        _basketService = basketService;
    }

    // [BindProperty] 讓 ASP.NET Core 在 POST 時
    // 自動把表單欄位的值填入這個屬性
    [BindProperty]
    public BasketCheckoutModel Order { get; set; } = default!;

    public async Task OnGetAsync()
    {
        // 取得購物車資訊，預填表單
        var basket = await _basketService.GetBasket("swn");
        Order = new BasketCheckoutModel
        {
            UserName = basket.UserName,
            TotalPrice = basket.TotalPrice
        };
    }

    public async Task<IActionResult> OnPostCheckOutAsync()
    {
        // Order 屬性已由框架自動填入表單送出的值
        if (!ModelState.IsValid) return Page();

        Order.CustomerId = new Guid("8bce5a2b-b494-4552-a3eb-290c87a2eb6a");
        await _basketService.CheckoutBasket(Order);

        return RedirectToPage("Confirmation", new { orderId = Order.UserName });
    }
}

ℹ️ [BindProperty] 的運作方式：表單送出（HTTP POST）時，ASP.NET Core 會掃描 PageModel 中標記 [BindProperty] 的屬性，並嘗試從表單欄位、路由值、查詢字串中對應填入。 屬性名稱 Order 會對應 name="Order.FirstName"、name="Order.LastName" 等表單欄位。

  • 了解這個服務目前缺少的安全性和 UX 功能
三個主要硬編碼問題
⚠️ 以下問題在學習專案中是刻意簡化的，實際生產環境必須改善。

#問題目前做法正式環境應改為
1 UserName 寫死 IBasketService.GetBasket("swn") 從 ASP.NET Core Identity 或 JWT Claims 取得：User.Identity?.Name
2 CustomerId 寫死 Order.CustomerId = new Guid("8bce...") 從 JWT Claim 取得：User.FindFirstValue("sub")
3 商品顏色預設 Color = "Black"（寫死在加入購物車邏輯） 加入顏色選擇器，從表單取得 selectedColor 參數
正式環境應加入的功能
正確取得當前使用者（參考做法）C#
public class IndexModel : PageModel
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    // 注入 IHttpContextAccessor 以存取當前使用者
    public IndexModel(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    private string GetCurrentUserName()
    {
        // ASP.NET Core Identity：取得登入的使用者名稱
        return _httpContextAccessor.HttpContext?.User.Identity?.Name
               ?? throw new UnauthorizedAccessException();
    }

    private Guid GetCurrentCustomerId()
    {
        // JWT Claim：取得 "sub" claim 作為 CustomerId
        var sub = _httpContextAccessor.HttpContext?.User
                      .FindFirstValue(ClaimTypes.NameIdentifier);
        return Guid.Parse(sub!);
    }
}
其他已知限制
安全性
  • 未強化 CSRF Token（雖然 Razor Pages 預設有 AntiForgery，但未驗證 AJAX 請求）
  • 無身份驗證機制（應加入 ASP.NET Core Identity 或 OpenID Connect）
  • 未授權使用者可直接訪問 /Cart /Checkout /OrderList
使用者體驗
  • 表單驗證錯誤沒有對應的 UI 提示（ModelState 錯誤被忽略）
  • 商品列表無分頁，大量資料時效能差
  • 無 Loading 狀態指示，API 呼叫期間畫面沒有回饋

完整資料流
瀏覽器 ↓ HTTP 請求（:6005 HTTP / :6065 HTTPS） Razor Pages（Shopping.Web） ↓ Refit HTTP Client（所有請求都指向同一個 GatewayAddress） YarpApiGateway（:6004 HTTP / :6064 HTTPS） ↓ 依路徑前綴路由（YARP Reverse Proxy） ├─ /catalog-service/* → Catalog.API（:6000） ├─ /basket-service/* → Basket.API（:6001） └─ /ordering-service/* → Ordering.API（:6003）

💡 Shopping.Web 不直接呼叫微服務的原因：
1. 單一入口：前端只需知道一個 Gateway 位址，不需維護多個 API 端點設定
2. 安全隔離：微服務不需暴露在公網，Gateway 可做認證/限流/轉換
3. 靈活性：後端服務可以遷移、重命名，只要更新 Gateway 設定即可，前端不受影響
各頁面職責
頁面路由職責使用的 Service
Index / 首頁商品列表，加入購物車 ICatalogService、IBasketService
ProductList /ProductList 依分類瀏覽商品 ICatalogService
ProductDetail /ProductDetail/{id} 單一商品詳情頁 ICatalogService
Cart /Cart 購物車檢視，可移除商品 IBasketService
Checkout /Checkout 填寫收件資訊並送出訂單 IBasketService
Confirmation /Confirmation 訂單送出後的確認頁面 無（靜態頁面）
OrderList /OrderList 查看歷史訂單 IOrderingService
