Repository Pattern

把資料存取邏輯從 Handler 抽出，透過介面隔離
學習目標
  • 理解為什麼 Basket 要用 Repository，而 Catalog 不用
  • 理解 Interface 在 Repository Pattern 中的角色：讓 Handler 不依賴具體實作
  • 理解 Marten 在 Basket 中如何以 UserName 作為自然主鍵
  • 理解 BasketNotFoundException 和 NotFoundException 的繼承關係
為什麼 Basket 需要 Repository
💡 Catalog 的 Handler 直接注入 IQuerySession，因為資料存取邏輯簡單。 Basket 的 Handler 注入 IBasketRepository，因為後面要在這個介面上加快取裝飾器（階段二）。 如果沒有介面，裝飾器就無從掛載。

public interface IBasketRepository
{
    Task<ShoppingCart> GetBasket(string userName, CancellationToken ct = default);
    Task<ShoppingCart> StoreBasket(ShoppingCart basket, CancellationToken ct = default);
    Task<bool> DeleteBasket(string userName, CancellationToken ct = default);
}

Data/BasketRepository.csC#
public class BasketRepository(IDocumentSession session) : IBasketRepository
{
    public async Task<ShoppingCart> GetBasket(string userName, CancellationToken ct = default)
    {
        var basket = await session.LoadAsync<ShoppingCart>(userName, ct);

        // 找不到就拋出例外，交給 CustomExceptionHandler 轉成 HTTP 404
        return basket is null
            ? throw new BasketNotFoundException(userName)
            : basket;
    }

    public async Task<ShoppingCart> StoreBasket(ShoppingCart basket, CancellationToken ct = default)
    {
        session.Store(basket);
        await session.SaveChangesAsync(ct);
        return basket;
    }

    public async Task<bool> DeleteBasket(string userName, CancellationToken ct = default)
    {
        session.Delete<ShoppingCart>(userName);
        await session.SaveChangesAsync(ct);
        return true;
    }
}

裝飾器模式說明
💡 不用裝飾器的做法：直接在 BasketRepository 裡加快取邏輯， 導致一個 class 同時負責資料庫存取和快取，違反單一職責原則。

裝飾器的做法：建立 CachedBasketRepository， 它包住 BasketRepository，在外層攔截呼叫，處理快取邏輯， 兩個 class 各自只做一件事。
實作快取裝飾器
Data/CachedBasketRepository.csC#
public class CachedBasketRepository(
    IBasketRepository repository,   // 注入被裝飾的原始 Repository
    IDistributedCache cache)         // 注入 Redis
    : IBasketRepository              // 實作同一個介面
{
    public async Task<ShoppingCart> GetBasket(string userName, CancellationToken ct = default)
    {
        // ① 先查 Redis
        var cached = await cache.GetStringAsync(userName, ct);
        if (cached is not null)
            return JsonSerializer.Deserialize<ShoppingCart>(cached)!;

        // ② Cache Miss：去資料庫拿
        var basket = await repository.GetBasket(userName, ct);

        // ③ 存進 Redis 供下次使用
        await cache.SetStringAsync(userName, JsonSerializer.Serialize(basket), ct);

        return basket;
    }

    public async Task<ShoppingCart> StoreBasket(ShoppingCart basket, CancellationToken ct = default)
    {
        // 寫入資料庫
        await repository.StoreBasket(basket, ct);

        // 同步更新 Redis
        await cache.SetStringAsync(basket.UserName, JsonSerializer.Serialize(basket), ct);

        return basket;
    }

    public async Task<bool> DeleteBasket(string userName, CancellationToken ct = default)
    {
        // 刪資料庫
        await repository.DeleteBasket(userName, ct);

        // 同步移除 Redis
        await cache.RemoveAsync(userName, ct);

        return true;
    }
}
用 Scrutor 串接，Handler 完全不需要改
Program.csC#
// Redis 設定
builder.Services.AddStackExchangeRedisCache(options =>
{
    options.Configuration = builder.Configuration.GetConnectionString("Redis");
});

// ① 先註冊原始實作
builder.Services.AddScoped<IBasketRepository, BasketRepository>();

// ② Scrutor 把 CachedBasketRepository 包在外面
// 當任何地方注入 IBasketRepository，實際上拿到的是：
// CachedBasketRepository（包著 BasketRepository（包著 IDocumentSession））
builder.Services.Decorate<IBasketRepository, CachedBasketRepository>();

// Handler 的程式碼完全不需要改動！

💡 這個 .proto 檔案是 Discount.Grpc 服務定義的，Basket.API 拿同一份來產生 Client。 雙方都用同一份合約，保證介面一致。

    string productName = 1;

在 Handler 中呼叫 gRPC
Basket/StoreBasket/StoreBasketCommandHandler.csC#
public class StoreBasketCommandHandler(
    IBasketRepository repository,
    DiscountProtoService.DiscountProtoServiceClient discountProto)  // gRPC Client
    : ICommandHandler<StoreBasketCommand, StoreBasketResult>
{
    public async Task<StoreBasketResult> Handle(
        StoreBasketCommand command, CancellationToken ct)
    {
        // ① 對購物車中每件商品呼叫 Discount 服務取得折扣
        await DeductDiscount(command.Cart, ct);

        // ② 儲存已套用折扣的購物車
        await repository.StoreBasket(command.Cart, ct);

        return new StoreBasketResult(command.Cart.UserName);
    }

    private async Task DeductDiscount(ShoppingCart cart, CancellationToken ct)
    {
        foreach (var item in cart.Items)
        {
            var coupon = await discountProto.GetDiscountAsync(
                new GetDiscountRequest { ProductName = item.ProductName },
                cancellationToken: ct);

            // 直接修改 item.Price，存進去的價格已是折扣後的
            item.Price -= coupon.Amount;
        }
    }
}
註冊 gRPC Client

builder.Services

⚠️ 同步呼叫的代價：購物車每次儲存，都必須對 Discount.Grpc 發出 N 次 gRPC 呼叫（N = 商品數量）。 如果 Discount 服務無回應，StoreBasket 整個請求也會失敗。 這是 Basket 和 Discount 之間的同步耦合，和 MassTransit 的非同步解耦形成對比。

  • 能說明同步 gRPC 和非同步 MassTransit 各自適合什麼場景

MassTransit + RabbitMQ

結帳時發布非同步事件，解耦 Basket 和 Order 服務
學習目標
  • 理解非同步訊息和同步 HTTP/gRPC 呼叫的根本差異
  • 理解 MassTransit 是什麼，以及它和 RabbitMQ 的關係
  • 理解 Publisher 和 Consumer 的角色，以及兩者如何透過訊息佇列解耦
  • 理解 CheckoutBasket 為什麼發布事件後立即刪除購物車
同步 vs 非同步通訊
🔗 同步（gRPC / REST）
呼叫方等待回應才繼續。
被呼叫方掛掉 → 呼叫方也失敗。
適合：需要即時結果（取得折扣金額）
📨 非同步（MassTransit / RabbitMQ）
發布訊息後立即繼續，不等回應。
Order 服務掛掉 → 訊息在佇列等候，不影響 Basket。
適合：不需要即時回應（結帳通知）

定義 Integration Event
💡 Integration Event 是跨服務邊界的事件，代表「某件事已經發生」。 和 Domain Event（服務內部）不同，Integration Event 需要能被序列化並透過訊息佇列傳遞。 在這個專案裡定義在 BuildingBlocks.Messaging，讓 Basket 和 Order 都可以引用同一個定義。

// 所有 Integration Event 的基底類別
public record IntegrationEvent
{
    public Guid Id => Guid.NewGuid();
    public DateTime OccurredOn => DateTime.Now;
    public string EventType => GetType().AssemblyQualifiedName!;
}

// Basket 發布，Order 訂閱
public record BasketCheckoutEvent : IntegrationEvent
{
    public string UserName { get; set; } = default!;
    public Guid CustomerId { get; set; }
    public decimal TotalPrice { get; set; }
    public string FirstName { get; set; } = default!;
    public string LastName { get; set; } = default!;
    public string EmailAddress { get; set; } = default!;
    // ... 地址、付款資訊欄位 ...
}
在 Handler 中發布事件
Basket/CheckoutBasket/CheckoutBasketCommandHandler.csC#
public class CheckoutBasketCommandHandler(
    IBasketRepository repository,
    IPublishEndpoint publishEndpoint)  // MassTransit 注入
    : ICommandHandler<CheckoutBasketCommand, CheckoutBasketResult>
{
    public async Task<CheckoutBasketResult> Handle(
        CheckoutBasketCommand command, CancellationToken ct)
    {
        // ① 取得現有購物車（包含 TotalPrice）
        var basket = await repository.GetBasket(command.BasketCheckoutDto.UserName, ct);

        // ② 把 DTO 對應成 Event，補上 TotalPrice
        var eventMessage = command.BasketCheckoutDto.Adapt<BasketCheckoutEvent>();
        eventMessage.TotalPrice = basket.TotalPrice;

        // ③ 發布到 RabbitMQ，Order 服務會收到並建立訂單
        // 這是非同步的：不等 Order 服務確認，直接繼續
        await publishEndpoint.Publish(eventMessage, ct);

        // ④ 購物車使命完成，刪除
        await repository.DeleteBasket(command.BasketCheckoutDto.UserName, ct);

        return new CheckoutBasketResult(true);
    }
}

為什麼事件定義在共用套件
💡 Publisher（Basket）和 Consumer（Order）必須對同一個訊息型別達成共識， 才能正確序列化和反序列化。

不好的做法：Basket 和 Order 各自定義一個名字相同但可能不同的 class，靠名稱字串比對。
好的做法：定義在 BuildingBlocks.Messaging，兩邊都引用同一份，編譯時就能保證一致。

和 Catalog 相比，Basket 多了哪些複雜度
💡 Basket 的複雜度來自它橫跨了三種通訊模式：

同步讀寫：使用者操作購物車（GET / POST / DELETE）→ PostgreSQL + Redis
同步呼叫：儲存時取折扣 → gRPC（需要即時結果）
非同步事件：結帳 → RabbitMQ（不需要即時結果，解耦服務）

三種模式都在同一個服務裡，是學習微服務通訊模式的最佳範本。
