認識 .proto 檔案語法
💡 .proto 是 gRPC 的「合約文件」，相當於 REST 的 OpenAPI spec。 差別在於 .proto 是強制的，而且可以直接從它產生 Server 和 Client 的程式碼， 不需要任何額外工具。

Protos/discount.protoprotobuf
syntax = "proto3";

// 產生的 C# 類別會放在這個 namespace
option csharp_namespace = "Discount.Grpc";

package discount;

// service：定義這個 gRPC 服務提供哪些方法
// 相當於 REST API 的 Controller
service DiscountProtoService {
    rpc GetDiscount    (GetDiscountRequest)    returns (CouponModel);
    rpc CreateDiscount (CreateDiscountRequest) returns (CouponModel);
    rpc UpdateDiscount (UpdateDiscountRequest) returns (CouponModel);
    rpc DeleteDiscount (DeleteDiscountRequest) returns (DeleteDiscountResponse);
}

// message：定義資料結構
// 相當於 C# 的 class / record，但欄位要標號碼（序列化用）
message GetDiscountRequest {
    string productName = 1;  // 欄位編號從 1 開始，不能改動（影響序列化）
}

message CouponModel {
    int32  id          = 1;
    string productName = 2;
    string description = 3;
    int32  amount      = 4;
}

message CreateDiscountRequest {
    CouponModel coupon = 1;  // message 可以包含其他 message
}

message UpdateDiscountRequest {
    CouponModel coupon = 1;
}

message DeleteDiscountRequest {
    string productName = 1;
}

message DeleteDiscountResponse {
    bool success = 1;
}

為什麼 Discount 用 SQLite 而不是 PostgreSQL
💡 Discount 服務只有簡單的折扣資料（商品名稱 + 折扣金額）， 資料量小、結構固定，不需要完整的資料庫伺服器。 SQLite 是單一檔案，啟動快、不需要額外的 Docker container， 非常適合這種內部輔助服務。

public class DiscountContext(DbContextOptions<DiscountContext> options)
    : DbContext(options)
{
    public DbSet<Coupon> Coupons => Set<Coupon>();

    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        // Seed Data：Migration 執行時自動插入
        modelBuilder.Entity<Coupon>().HasData(

            new Coupon { Id = 2, ProductName = "Samsung 10", Amount = 100, Description = "Samsung 10 Discount" }
        );
    }
}

實作 gRPC Service

繼承 proto 產生的 base class，覆寫四個 CRUD 方法
學習目標
  • 理解 gRPC Service 的實作方式：繼承 base class、覆寫方法
  • 理解 ServerCallContext 的角色（相當於 REST 的 HttpContext）
  • 理解 RpcException 和 StatusCode 的對應關係（類似 HTTP 狀態碼）
  • 理解 Mapster 在 Coupon（EF Core entity）和 CouponModel（proto message）之間的轉換
實作 DiscountService
Services/DiscountService.csC#
// 繼承 proto 自動產生的 base class
public class DiscountService(
    DiscountContext dbContext,
    ILogger<DiscountService> logger)
    : DiscountProtoService.DiscountProtoServiceBase
{
    // ServerCallContext 包含 gRPC 呼叫的中繼資料（header、deadline、cancellation token 等）
    // 相當於 REST Controller 的 HttpContext，但通常不需要直接存取

    public override async Task<CouponModel> GetDiscount(
        GetDiscountRequest request, ServerCallContext context)
    {
        var coupon = await dbContext.Coupons
            .FirstOrDefaultAsync(c => c.ProductName == request.ProductName);

        // 找不到不拋例外，回傳「無折扣」的預設值
        // 這樣 Basket 在呼叫時不需要處理例外情況
        coupon ??= new Coupon
        {
            ProductName = "No Discount",
            Amount = 0,
            Description = "No Discount Desc"
        };

        logger.LogInformation("Discount is retrieved for ProductName: {ProductName}, Amount: {Amount}",
            coupon.ProductName, coupon.Amount);

        // Coupon（EF Core entity）→ CouponModel（proto message）
        return coupon.Adapt<CouponModel>();
    }

    public override async Task<CouponModel> CreateDiscount(
        CreateDiscountRequest request, ServerCallContext context)
    {
        // proto message → EF Core entity
        var coupon = request.Coupon.Adapt<Coupon>()
            ?? throw new RpcException(new Status(StatusCode.InvalidArgument, "Invalid request"));
        // RpcException + StatusCode.InvalidArgument ≈ HTTP 400 Bad Request

        dbContext.Coupons.Add(coupon);
        await dbContext.SaveChangesAsync();

        logger.LogInformation("Discount is successfully created. ProductName: {ProductName}",
            coupon.ProductName);

        return coupon.Adapt<CouponModel>();
    }

    public override async Task<CouponModel> UpdateDiscount(
        UpdateDiscountRequest request, ServerCallContext context)
    {
        var coupon = request.Coupon.Adapt<Coupon>()
            ?? throw new RpcException(new Status(StatusCode.InvalidArgument, "Invalid request"));

        dbContext.Coupons.Update(coupon);
        await dbContext.SaveChangesAsync();

        logger.LogInformation("Discount is successfully updated. ProductName: {ProductName}",
            coupon.ProductName);

        return coupon.Adapt<CouponModel>();
    }

    public override async Task<DeleteDiscountResponse> DeleteDiscount(
        DeleteDiscountRequest request, ServerCallContext context)
    {
        var coupon = await dbContext.Coupons
            .FirstOrDefaultAsync(c => c.ProductName == request.ProductName)
            ?? throw new RpcException(new Status(StatusCode.NotFound,
                $"Discount with ProductName={request.ProductName} not found"));
        // RpcException + StatusCode.NotFound ≈ HTTP 404 Not Found

        dbContext.Coupons.Remove(coupon);
        await dbContext.SaveChangesAsync();

        logger.LogInformation("Discount is successfully deleted. ProductName: {ProductName}",
            request.ProductName);

        return new DeleteDiscountResponse { Success = true };
    }
}

在 Handler 中使用 gRPC Client
Basket — StoreBasketCommandHandler.cs（關鍵部分）C#
public class StoreBasketCommandHandler(
    IBasketRepository repository,
    DiscountProtoService.DiscountProtoServiceClient discountProto)
    : ICommandHandler<StoreBasketCommand, StoreBasketResult>
{
    private async Task DeductDiscount(ShoppingCart cart, CancellationToken ct)
    {
        foreach (var item in cart.Items)
        {
            // 對每件商品同步呼叫 Discount.Grpc
            // GetDiscountAsync：非同步版本（效能較好）
            // GetDiscount：同步版本（阻塞執行緒）
            var coupon = await discountProto.GetDiscountAsync(
                new GetDiscountRequest { ProductName = item.ProductName },
                cancellationToken: ct);

            // coupon.Amount 是 int，item.Price 是 decimal，需要型別轉換
            item.Price -= coupon.Amount;
        }
    }
}

為什麼沒有 MediatR 和 CQRS
💡 Discount 服務只有四個簡單的 CRUD 操作，業務邏輯幾乎沒有。 加入 MediatR + CQRS 反而會增加不必要的複雜度（多出 Command、Query、Handler 等類別）。 架構應該配合需求選用，不是所有服務都需要相同的複雜度。 這正體現了微服務架構的彈性：每個服務可以根據自身需求選擇最適合的技術棧。
