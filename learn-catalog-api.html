定義 Query 和 Command
💡 Query 代表讀取資料（不改變狀態），Command 代表寫入資料（改變狀態）。 這個區分是 CQRS 的核心概念。

// Query：代表「我想讀取商品列表」這個意圖
// IRequest<T> 告訴 MediatR：執行這個 Request 後會回傳 T
public record GetProductsQuery() : IRequest<List<ProductDto>>;

public record ProductDto(Guid Id, string Name, decimal Price);

// Command：代表「我想建立一個商品」這個意圖
public record CreateProductCommand(string Name, decimal Price)
    : IRequest<Guid>;  // 回傳新商品的 ID

註冊並使用
Program.csC#
var builder = WebApplication.CreateBuilder(args);

// 告訴 MediatR 去掃描這個 Assembly 裡所有的 IRequestHandler
builder.Services.AddMediatR(cfg =>
    cfg.RegisterServicesFromAssemblyContaining<Program>());

var app = builder.Build();

// Endpoint 只依賴 ISender，完全不知道 Handler 的存在
app.MapGet("/products", async (ISender sender) =>
{
    var result = await sender.Send(new GetProductsQuery());
    return Results.Ok(result);
});

app.MapPost("/products", async (CreateProductCommand command, ISender sender) =>
{
    var id = await sender.Send(command);
    return Results.Created($"/products/{id}", new { id });
});

app.Run();
💡 注意 Endpoint 裡完全看不到 GetProductsHandler 或 CreateProductHandler。 MediatR 根據 Request 的型別自動找到對應的 Handler。這就是解耦的意義。

  • 能說明 ISender.Send() 和直接呼叫 Handler 的差別

FluentValidation + Pipeline Behavior

安裝套件
Terminalshell

定義 Validator
💡 每個 Command 或 Query 都可以有自己的 Validator。 Validator 和 Handler 放在同一個資料夾，這正是 Vertical Slice 的精神。

Commands/CreateProductValidator.csC#
public class CreateProductCommandValidator : AbstractValidator<CreateProductCommand>
{
    public CreateProductCommandValidator()
    {
        RuleFor(x => x.Name)
            .NotEmpty().WithMessage("商品名稱不能為空")
            .MinimumLength(2).WithMessage("商品名稱至少 2 個字")
            .MaximumLength(100).WithMessage("商品名稱最多 100 個字");

        RuleFor(x => x.Price)
            .GreaterThan(0).WithMessage("價格必須大於 0");
    }
}
實作 Pipeline Behavior
💡 Pipeline Behavior 就像是 Handler 的「前後處理器」。 執行順序為：Behavior 前段 → Handler → Behavior 後段。 可以有多個 Behavior 串接，形成一條 Pipeline。

Behaviors/ValidationBehavior.csC#
public class ValidationBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : IRequest<TResponse>
{
    private readonly IEnumerable<IValidator<TRequest>> _validators;

    public ValidationBehavior(IEnumerable<IValidator<TRequest>> validators)
        => _validators = validators;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,  // next() 呼叫下一個 Behavior 或 Handler
        CancellationToken cancellationToken)
    {
        // ① 執行所有 Validator
        var context = new ValidationContext<TRequest>(request);
        var failures = _validators
            .Select(v => v.Validate(context))
            .SelectMany(r => r.Errors)
            .Where(f => f != null)
            .ToList();

        // ② 有錯誤就拋出例外，Handler 不會被執行
        if (failures.Any())
            throw new ValidationException(failures);

        // ③ 沒有錯誤，繼續執行 Handler
        return await next();
    }
}
註冊並處理驗證錯誤
Program.csC#
builder.Services.AddMediatR(cfg =>
{
    cfg.RegisterServicesFromAssemblyContaining<Program>();
    // 把 ValidationBehavior 加入 Pipeline
    cfg.AddOpenBehavior(typeof(ValidationBehavior<,>));
});

// 自動掃描並註冊所有 AbstractValidator
builder.Services.AddValidatorsFromAssemblyContaining<Program>();

// 全域例外處理：把 ValidationException 轉成 HTTP 400
app.UseExceptionHandler(errApp =>
{
    errApp.Run(async ctx =>
    {
        var ex = ctx.Features.Get<IExceptionHandlerFeature>()?.Error;
        if (ex is ValidationException ve)
        {
            ctx.Response.StatusCode = 400;
            await ctx.Response.WriteAsJsonAsync(new
            {
                errors = ve.Errors.Select(e => new { e.PropertyName, e.ErrorMessage })
            });
        }
    });
});

💡 Marten 不需要定義資料表結構。只要是一個 C# class，加上 Id 屬性， Marten 就會自動把它序列化成 JSON 存進 PostgreSQL，Schema 自動建立。

Models/Product.csC#
public class Product
{
    public Guid Id { get; set; }
    public string Name { get; set; } = default!;
    public decimal Price { get; set; }
    public List<string> Category { get; set; } = [];

    // 注意：不需要 [Key]、不需要 DbContext、不需要 DbSet

更新 Handler 使用 Marten
GetProductsHandler.csC#
public class GetProductsHandler
    : IRequestHandler<GetProductsQuery, List<ProductDto>>
{
    private readonly IQuerySession _session;

    public GetProductsHandler(IQuerySession session)
        => _session = session;

    public async Task<List<ProductDto>> Handle(
        GetProductsQuery request,
        CancellationToken ct)
    {
        var products = await _session
            .Query<Product>()
            .ToListAsync(ct);

        return products
            .Select(p => new ProductDto(p.Id, p.Name, p.Price))
            .ToList();
    }
}
CreateProductHandler.csC#
public class CreateProductHandler
    : IRequestHandler<CreateProductCommand, Guid>
{
    private readonly IDocumentSession _session;

    public CreateProductHandler(IDocumentSession session)
        => _session = session;

    public async Task<Guid> Handle(
        CreateProductCommand request,
        CancellationToken ct)
    {
        var product = new Product
        {
            Id = Guid.NewGuid(),
            Name = request.Name,
            Price = request.Price
        };

        // Store 是在記憶體中標記，SaveChanges 才真正寫入
        _session.Store(product);
        await _session.SaveChangesAsync(ct);

        return product.Id;
    }
}
IQuerySession vs IDocumentSession
📖 IQuerySession
只讀。比 IDocumentSession 輕量，用於 Query Handler。
✏️ IDocumentSession
可讀寫。有 Store / Delete / SaveChanges，用於 Command Handler。

對照清單
你寫的 vs Catalog.API對照
// ① 你的 IRequest — Catalog 用更嚴格的 ICommand / IQuery 包裝
// 你的寫法：
public record GetProductsQuery() : IRequest<List<ProductDto>>;

// Catalog 的寫法（BuildingBlocks/CQRS）：
public record GetProductsQuery(...) : IQuery<GetProductsResult>;
// IQuery<T> 繼承自 IRequest<T>，加了 [NotNull] 限制，語意更清楚

// ② 你的 Handler — Catalog 用 IQueryHandler / ICommandHandler
// 你的寫法：
public class GetProductsHandler : IRequestHandler<GetProductsQuery, List<ProductDto>>

// Catalog 的寫法：
public class GetProductsHandler : IQueryHandler<GetProductsQuery, GetProductsResult>
// IQueryHandler 是 IRequestHandler 的別名，只是讓意圖更明顯

// ③ 你的 Endpoint 直接用 Command — Catalog 用 Mapster 轉換
// 你的寫法：
app.MapPost("/products", async (CreateProductCommand command, ISender sender) => ...);

// Catalog 的寫法：
app.MapPost("/products", async (CreateProductRequest request, ISender sender) =>
{
    var command = request.Adapt<CreateProductCommand>(); // Mapster
    var result = await sender.Send(command);
    return Results.Created(...);
});
// 分開 Request DTO 和 Command，讓兩者可以獨立演進
BuildingBlocks 的設計原因
💡 你在階段一到四寫的 ValidationBehavior、自訂例外、CQRS 介面， 在 Catalog.API 裡被抽到 BuildingBlocks 專案， 原因是這些東西在 Basket.API、Ordering.API 裡一模一樣， 共用而不是複製貼上。微服務共用的基礎設施應該獨立成一個套件。

Behaviors/LoggingBehavior.csC#
public class LoggingBehavior<TRequest, TResponse>
    : IPipelineBehavior<TRequest, TResponse>
    where TRequest : notnull
{
    private readonly ILogger<LoggingBehavior<TRequest, TResponse>> _logger;

    public LoggingBehavior(ILogger<LoggingBehavior<TRequest, TResponse>> logger)
        => _logger = logger;

    public async Task<TResponse> Handle(
        TRequest request,
        RequestHandlerDelegate<TResponse> next,
        CancellationToken cancellationToken)
    {
        _logger.LogInformation("[START] {Request}", typeof(TRequest).Name);
        var sw = Stopwatch.StartNew();

        var response = await next();

        sw.Stop();
        if (sw.ElapsedMilliseconds > 3000)
            _logger.LogWarning("[SLOW] {Request} took {Ms}ms",
                typeof(TRequest).Name, sw.ElapsedMilliseconds);

        _logger.LogInformation("[END] {Request} ({Ms}ms)",
            typeof(TRequest).Name, sw.ElapsedMilliseconds);

        return response;
    }
}

// Program.cs 加入 Pipeline（順序很重要：Logging → Validation → Handler）
cfg.AddOpenBehavior(typeof(LoggingBehavior<,>));
