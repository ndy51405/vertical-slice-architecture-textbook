YARP 基本概念

Reverse Proxy、Route、Cluster 的本質
學習目標
  • 理解 Reverse Proxy 是什麼，以及它在微服務架構中的角色
  • 理解為什麼微服務需要 API Gateway，而不是讓客戶端直接呼叫各服務
  • 比較 YARP vs Nginx vs Ocelot 三種方案的差異與適用情境
  • 理解 Route 和 Cluster 的關係：Route 描述「如何比對請求」，Cluster 描述「轉發到哪裡」
為什麼需要 API Gateway？

在微服務架構中，可能有十幾個服務各自監聽不同的 Port。如果客戶端（瀏覽器、App）直接呼叫每個服務，會面臨以下問題：

⚠
沒有 Gateway 的問題
客戶端需要知道所有服務的位址；跨域 (CORS) 設定複雜；無法統一做認證、限流、日誌；服務位址變動時客戶端需跟著更新。
✓
有了 Gateway 的好處
統一入口（單一 Host/Port）；集中認證與授權；統一 Rate Limiting；服務位址對客戶端透明；方便做 A/B 測試、藍綠部署。
YARP vs Nginx vs Ocelot
比較項目 YARP Nginx Ocelot
設定方式 appsettings.json 或程式碼（C#） nginx.conf（專屬語法） ocelot.json（專屬 JSON）
C# 整合 原生 ASP.NET Core Middleware，可用 C# 擴充 無（需用 Lua 或外部模組） 可用 C# 擴充，但 API 較舊
效能 高（基於 .NET Kestrel，直接 Pipe） 極高（C 語言核心） 中（.NET，但轉發效率較低）
熱更新設定 支援（IProxyConfigProvider） 需 reload signal 需重啟或自訂
維護狀態 Microsoft 官方維護（活躍） F5 維護（非常成熟） 社群維護（更新較慢）
適合情境 .NET 微服務需要程式邏輯的 Gateway 高流量、靜態資源、傳統架構 早期 .NET Core 微服務專案

Terminal shell 
# 建立最小 Web 專案
dotnet new web -n LearnYarp
cd LearnYarp

# 安裝 YARP NuGet 套件
dotnet add package Yarp.ReverseProxy
Program.cs 最小設定
i
YARP 以標準 ASP.NET Core Middleware 方式運作。只需兩行：AddReverseProxy() 註冊服務，MapReverseProxy() 掛載 Middleware。
Program.cs C# 
var builder = WebApplication.CreateBuilder(args);

// 註冊 YARP 服務，並從 appsettings.json 的 ReverseProxy 段落載入設定
builder.Services
    .AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

var app = builder.Build();

// 掛載 YARP Middleware，所有符合 Route 設定的請求都會被轉發
app.MapReverseProxy();

app.Run();
i
注意：MapReverseProxy() 必須在 app.Run() 之前呼叫。如果有其他 Middleware（如 Rate Limiter），需注意順序，Rate Limiter 必須在 MapReverseProxy() 之前呼叫 UseRateLimiter()。
appsettings.json 最小設定
appsettings.json JSON 
{
  "ReverseProxy": {
    "Routes": {
      "catalog-route": {                    // Route 名稱（任意字串）
        "ClusterId": "catalog-cluster",     // 對應到下方 Clusters 的哪個
        "Match": {
          "Path": "/catalog-service/{**catch-all}"  // 比對路徑
        },
        "Transforms": [
          { "PathRemovePrefix": "/catalog-service" }  // 轉發前移除前綴
        ]
      }
    },
    "Clusters": {
      "catalog-cluster": {              // Cluster 名稱
        "Destinations": {
          "destination1": {              // Destination 名稱（可有多個）
            "Address": "http://localhost:6000/"
          }
        }
      }
    }
  }
}
關鍵概念說明
★
{**catch-all} 萬用符號
這是 ASP.NET Core Route 語法。** 表示「比對任意路徑片段，包含斜線」。例如 /catalog-service/products/123 和 /catalog-service/categories/food/list 都會被這條 Route 比對到。單個 * 不能跨斜線，必須用 **。
★
PathRemovePrefix 轉換
客戶端請求 /catalog-service/products，YARP 移除前綴後，轉發到後端的 /products。這讓後端服務不需要知道自己被掛在哪個路徑前綴下，保持乾淨的 API 設計。
i
Cluster 可以有多個 Destination
一個 Cluster 底下可以定義多個 Destination，YARP 會根據 LoadBalancingPolicy（預設 RoundRobin）在多個目標之間分配請求，實現水平擴展與負載平衡。
⚠
Route vs Cluster 的關係
Route = 「什麼樣的請求要被處理」（比對規則）。Cluster = 「轉發到哪裡」（目標清單）。一個 Route 對應一個 Cluster，但一個 Cluster 可以被多個 Route 共用。

Docker 服務名稱 vs localhost 的差異

Docker Compose 環境

# docker-compose.yml 中定義的服務名稱
services:
  catalog.api:
    image: catalog-api
    ports: ["6000:8080"]

  yarp.gateway:
    image: yarp-gateway
    ports: ["6004:8080"]

在 Docker 網路中，容器之間用服務名稱 catalog.api 互相溝通，Port 是容器內部 Port（8080），不是宿主機映射的 Port（6000）。

本地開發環境

# 本地直接跑各服務，各自監聽 localhost
# Catalog.API → localhost:6000
# Basket.API  → localhost:6001
# Ordering.API → localhost:6003
# YarpGateway → localhost:6004

本地直接執行時，所有服務都在同一台機器，用 localhost:{port} 存取。沒有 Docker DNS，所以 catalog.api 這個名稱無法解析。

i
為什麼 appsettings.json 只覆蓋 Clusters 而不覆蓋 Routes？
Routes 的比對規則（Path Pattern）在兩個環境是一樣的，不需要改。只有 Cluster 中的 Destination Address（目標位址）不同，所以 appsettings.Local.json 只覆寫 Clusters 段落即可。

Rate Limiting

固定視窗演算法、保護特定服務的流量
學習目標
  • 理解 Rate Limiting（速率限制）的用途：防止濫用、保護後端服務資源
  • 理解固定視窗（Fixed Window）演算法的運作方式
  • 理解為什麼只對 Ordering 服務套用限流，而 Catalog 不需要
  • 理解如何在 Route 設定中指定 RateLimiterPolicy
固定視窗 vs 滑動視窗
★
固定視窗（Fixed Window）
將時間切成等長的「窗口」（例如每 10 秒一格）。每個窗口內最多允許 N 次請求。窗口結束後重置計數。實作簡單，但在窗口邊界可能允許短暫的 2N 個請求（邊界效應）。
★
滑動視窗（Sliding Window）
以「過去 N 秒」為範圍動態計算請求數，沒有邊界效應，流量更平滑。但實作較複雜，需要記錄每個請求的時間戳。.NET 的 AddSlidingWindowLimiter 可以實作此演算法。
Program.cs 完整 Rate Limiting 設定
Program.cs C# 
var builder = WebApplication.CreateBuilder(args);

builder.Services
    .AddReverseProxy()
    .LoadFromConfig(builder.Configuration.GetSection("ReverseProxy"));

// 註冊 Rate Limiter 服務，定義 "fixed" Policy
builder.Services.AddRateLimiter(rateLimiterOptions =>
    rateLimiterOptions.AddFixedWindowLimiter(
        policyName: "fixed",
        options =>
        {
            // 時間窗口大小：10 秒
            options.Window = TimeSpan.FromSeconds(10);

            // 每個窗口最多允許 5 次請求
            options.PermitLimit = 5;

            // 超過限制時排隊等待的上限（0 = 直接拒絕）
            options.QueueLimit = 0;
        }));

var app = builder.Build();

// 必須在 MapReverseProxy 之前呼叫 UseRateLimiter！
app.UseRateLimiter();

app.MapReverseProxy();

app.Run();
⚠
Middleware 順序非常重要
UseRateLimiter() 必須在 MapReverseProxy() 之前呼叫，否則 Rate Limiter Middleware 不會在轉發之前執行，限流設定將無效。
appsettings.json 對 ordering-route 套用 Policy
appsettings.json（ordering-route 加上 RateLimiterPolicy） JSON 
{
  "ReverseProxy": {
    "Routes": {
      "catalog-route": {
        "ClusterId": "catalog-cluster",
        "Match": { "Path": "/catalog-service/{**catch-all}" },
        "Transforms": [{ "PathRemovePrefix": "/catalog-service" }]
        // 沒有 RateLimiterPolicy → 不限流
      },
      "basket-route": {
        "ClusterId": "basket-cluster",
        "Match": { "Path": "/basket-service/{**catch-all}" },
        "Transforms": [{ "PathRemovePrefix": "/basket-service" }]
        // 沒有 RateLimiterPolicy → 不限流
      },
      "ordering-route": {
        "ClusterId": "ordering-cluster",
        "Match": { "Path": "/ordering-service/{**catch-all}" },
        "Transforms": [{ "PathRemovePrefix": "/ordering-service" }],
        "RateLimiterPolicy": "fixed"  // 套用 "fixed" Policy
      }
    }
  }
}
為什麼只對 Ordering 限流？
i
Catalog 不需要限流的原因：商品列表查詢是唯讀操作，通常有快取機制（Redis、CDN）。高頻率的查詢不會造成資料狀態問題，後端通常也有資料庫讀取副本可以水平擴展。
⚠
Ordering 需要限流的原因：建立訂單是寫入操作，涉及庫存扣除、付款流程。惡意或意外的高頻請求可能造成重複下單、庫存超賣、資料庫鎖定問題。業務上也需要防止惡意搶購腳本。
i
超過限制時的回應：當請求超過 Rate Limiting 設定（10 秒內 5 次），YARP 會直接回傳 HTTP 429 Too Many Requests，並在 Response Header 中加入 Retry-After，告知客戶端何時可以重試。

進階功能

負載平衡、認證、健康檢查、請求轉換（說明為主）
學習目標
  • 知道 YARP 還支援哪些進階能力，以便未來需要時能快速找到對應設定
  • 了解負載平衡的多種演算法及其適用情境
  • 了解如何在 Gateway 層面做認證與授權（而非在每個服務重複實作）
  • 了解健康檢查機制如何讓 YARP 自動排除不健康的 Destination
負載平衡（Load Balancing）
i
只要在 Cluster 加入 LoadBalancingPolicy 並設定多個 Destination，YARP 就會自動在它們之間分配請求。
appsettings.json（RoundRobin 兩個 Destination） JSON 
{
  "ReverseProxy": {
    "Clusters": {
      "catalog-cluster": {
        "LoadBalancingPolicy": "RoundRobin",
        "Destinations": {
          "destination1": { "Address": "http://catalog.api-1:8080/" },
          "destination2": { "Address": "http://catalog.api-2:8080/" }
        }
      }
    }
  }
}
Policy 名稱 演算法說明 適用情境
RoundRobin 輪流分配，每個 Destination 依序各得一次 各服務效能相近，請求處理時間短
LeastRequests 優先選擇目前進行中請求數最少的 Destination 請求處理時間差異大（有的快有的慢）
Random 隨機選擇 簡單場景，不在乎分配均勻性
PowerOfTwoChoices 隨機選兩個，取請求數較少的那個 大規模叢集，比 LeastRequests 效率更高
認證與授權（Authentication & Authorization）
appsettings.json（Route 加上 AuthorizationPolicy） JSON 
{
  "ReverseProxy": {
    "Routes": {
      "ordering-route": {
        "ClusterId": "ordering-cluster",
        "AuthorizationPolicy": "default",  // 需要已認證的使用者
        "Match": { "Path": "/ordering-service/{**catch-all}" },
        "Transforms": [{ "PathRemovePrefix": "/ordering-service" }]
      }
    }
  }
}
i
設定 AuthorizationPolicy 後，未認證的請求會在 Gateway 層被攔截（回傳 401），不會到達後端服務。後端服務可以專注在業務邏輯，不需重複實作認證檢查。
健康檢查（Health Checks）
appsettings.json（Cluster 加上 HealthCheck） JSON 
{
  "ReverseProxy": {
    "Clusters": {
      "catalog-cluster": {
        "HealthCheck": {
          "Active": {
            "Enabled": true,
            "Interval": "00:00:10",    // 每 10 秒主動探測
            "Timeout": "00:00:05",    // 5 秒無回應視為不健康
            "Policy": "ConsecutiveFailures",
            "Path": "/health"          // 後端服務的健康檢查端點
          }
        },
        "Destinations": {
          "destination1": { "Address": "http://catalog.api:8080/" },
          "destination2": { "Address": "http://catalog.api-2:8080/" }
        }
      }
    }
  }
}
i
YARP 定期對每個 Destination 的 /health 路徑發送 GET 請求。若回傳非 2xx 狀態碼或超時，YARP 會自動將該 Destination 標記為不健康，新的請求不再路由到它，直到它恢復健康。
請求轉換（Request Transforms）
appsettings.json（多種 Transform 範例） JSON 
{
  "ReverseProxy": {
    "Routes": {
      "catalog-route": {
        "ClusterId": "catalog-cluster",
        "Match": { "Path": "/catalog-service/{**catch-all}" },
        "Transforms": [
          // 移除路徑前綴
          { "PathRemovePrefix": "/catalog-service" },

          // 加入自訂 Request Header，後端可讀取
          { "RequestHeader": "X-Gateway-Name", "Set": "YarpGateway" },

          // 將 X-Forwarded-For 傳遞給後端
          { "X-Forwarded": "Set" },

          // 修改 QueryString：加入固定參數
          { "QueryValueParameter": "source", "Set": "gateway" }
        ]
      }
    }
  }
}

★
為什麼 YarpApiGateway 只有兩個主要檔案？
這個服務本身沒有任何業務邏輯——它只負責「接收請求、查找 Route、轉發到對應 Cluster」。所有的行為都由 YARP 套件處理，開發者只需要提供設定（appsettings.json）。Program.cs 只需要告訴 ASP.NET Core「使用 YARP」，沒有 Controller、Handler、Repository 等元件。
